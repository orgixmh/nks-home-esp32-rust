NKS Home ESP32 Rust Backend

This document explains how the current firmware works by reading the Rust source code in src/. It is written for engineers who need to understand boot flow, runtime state, MQTT topics, provisioning, schema registry behavior, resource validation, logical devices, and the public backend contract well enough to build tools or a frontend.

Entry Point src/main.rs creates the NVS partition and runs AppController.
Main Modes The firmware runs in Normal mode when Wi-Fi and MQTT config are complete, otherwise in Provisioning.
Public Contract The frontend should rely on retained config snapshots and public dev/<id>/state and dev/<id>/cmd topics.
Current Runtime Driver Both built-in module types currently resolve to the same runtime driver: relay_gpio.

1. System Architecture

The firmware is organized around a strict split between configuration, runtime modules, logical devices, and MQTT contract publishing.

Board Layer src/board/mod.rs defines the target board profile and GPIO capabilities.
Config Layer src/config/types.rs defines the canonical schema-id-based config model.
Schema Layer src/schemas/* defines built-in module, device, and binding role schemas plus validation rules.
GPIO Layer GpioManager validates bindings, board support, and pin uniqueness, then claims GPIO pins for runtime use.
Module Layer ModuleManager builds runtime module instances from schemas and module config.
Device Layer DeviceRegistry builds logical devices from device config and bridges public device commands/states to runtime modules.
MQTT Layer MqttContract defines topic names, retained config publication, and config command parsing.
Provisioning Layer src/http/mod.rs serves the captive portal and staged Wi-Fi/MQTT test-and-save flow.
The current design intentionally uses schema IDs everywhere in the config contract: module_type_id, device_type_id, and role_id.

2. Boot And Runtime Flow

Startup Sequence

  1. main() initializes ESP-IDF patches and logging, takes the default NVS partition, and runs AppController.
  2. AppController::new() creates: the config store, board-specific GPIO manager, built-in schema registry, LED indicator, and runtime state holder.
  3. AppController::run(): sets the LED to Off, clears hardcoded demo seed config if present, loads and validates resource config, then selects Normal or Provisioning mode.

Boot Mode Decision

detect_boot_mode() in src/app/mod.rs loads the stored config and checks DeviceConfig::is_complete().

  • Normal: Wi-Fi and MQTT fields are present and non-empty.
  • Provisioning: anything missing or incomplete.

Normal Mode Flow

  1. Load the full stored config and normalize resources.
  2. Create ESP peripherals and system event loop.
  3. Create Wi-Fi station mode and connect with retry.
  4. Build runtime modules from module_instances.
  5. Build logical devices from device_instances.
  6. Connect to MQTT and subscribe to contract, device, and module command topics.
  7. Publish retained birth/config snapshots.
  8. Start the local HTTP server used in normal operation.
  9. Enter the operational loop.

Operational Loop Responsibilities

  • Watch Wi-Fi connectivity and retry if the station link drops.
  • Watch MQTT connectivity and wait/reconnect if the broker session drops.
  • Poll module drivers for timer/input-triggered changes.
  • Publish changed module states on internal mod/... topics.
  • Publish changed logical device states on public dev/.../state topics.
  • Handle incoming MQTT config commands, logical device commands, and internal module commands.

Provisioning Mode Flow

  1. Start a provisioning access point with a deterministic SSID based on AP MAC address.
  2. Scan nearby Wi-Fi networks.
  3. Start the captive portal server.
  4. Wait for Wi-Fi test requests and MQTT test requests from the provisioning UI.
  5. Persist tested Wi-Fi and MQTT config into NVS.
  6. Trigger reboot once provisioning is complete.

3. Runtime States And LED Indicator

Runtime state is represented by RuntimeState in src/runtime/mod.rs. It is made of a status, an optional degraded reason, and an optional current action.

Field Values Meaning
status Provisioning, Operational, Degraded High-level current mode of the controller.
reason WifiDisconnected, MqttDisconnected, Unknown Why the system is degraded.
action WifiConnecting, MqttConnecting, or none What the runtime is actively trying to do right now.

State Transitions

  • set_operational_action(): status is Operational and the runtime is actively connecting.
  • set_provisioning_action(): status is Provisioning and the provisioning flow is actively testing Wi-Fi or MQTT.
  • set_degraded(): status is Degraded with an explicit reason and optional action.

LED Indicator Behavior

The indicator implementation lives in src/runtime/indicator.rs. It runs in its own thread and watches a shared IndicatorMode.

Condition LED Pattern
Off LED low
Provisioning Blink 120ms on / 120ms off
Operational with no action LED steady on
Operational while connecting Wi-Fi or MQTT Blink 500ms on / 500ms off
Degraded because Wi-Fi is disconnected 2 seconds off, then 2 blinks
Degraded because MQTT is disconnected 2 seconds off, then 4 blinks
Degraded unknown reason 2 seconds off, then 8 blinks

4. Configuration Model And NVS Storage

Top-Level Config Objects

The canonical config model is defined in src/config/types.rs.

WifiConfig Station SSID and password.
MqttConfig Broker host, port, credentials, MQTT client ID, and base topic.
ResourceConfig Schema-based module and device configuration with explicit version.
DeviceConfig Container for Wi-Fi, MQTT, and resources.

Canonical Resource Model

{
  "version": 1,
  "module_instances": [
    {
      "id": "relay1",
      "module_type_id": "core:gpio_switch",
      "display_name": "Kitchen Relay",
      "bindings": [
        {
          "role_id": "core:relay_output",
          "target": { "kind": "gpio", "pin": 4 }
        }
      ],
      "settings": {
        "auto_off_ms": null,
        "external_on_triggers": []
      }
    }
  ],
  "device_instances": [
    {
      "id": "relay1__device",
      "device_type_id": "core:switch",
      "display_name": "Kitchen Relay",
      "driver_module_id": "relay1"
    }
  ]
}

NVS Namespaces

src/storage/nvs.rs stores config in three namespaces:

Namespace Purpose Important Keys
wifi Wi-Fi station config ssid, pass
mqtt MQTT broker config host, port, user, pass, cid, topic
resources Full resource config JSON bindings
The backend currently includes a narrow compatibility bridge while deserializing older stored configs. It accepts old field names like module_type, device_type, and role, then maps them into schema IDs.

5. Schema Registry

The schema registry is the source of truth for what module types, device types, and binding roles exist. It lives in src/schemas/registry.rs, with built-ins defined in src/schemas/builtins.rs.

Current Built-In Binding Roles

core:relay_output
core:wall_trigger_input
core:output

Current Built-In Module Types

ID Display Name Binding Mode Default Device Type Runtime Driver
core:gpio_switch GPIO Switch single core:switch relay_gpio
core:gpio_output GPIO Output single core:switch relay_gpio

Current Built-In Device Types

ID Display Name Commands State Fields Capabilities
core:switch Switch ON, OFF, TOGGLE state (string) binary_output, retained_state

What The Registry Validates

  • Every referenced binding role exists.
  • Output binding and optional trigger binding are allowed by the module schema.
  • Every compatible device type exists.
  • A default device type, if present, exists and is compatible with the module schema.
  • A multi-device schema cannot declare a single default device type.

Schema Snapshots Published To MQTT

The registry generates deterministic snapshots for the frontend:

  • module_type_snapshots(): sorted by module type ID.
  • device_type_snapshots(): sorted by device type ID.

6. GPIO, Modules, And Devices

Board Layer

BoardProfile::esp32_devkit_v1() in src/board/mod.rs defines the supported pins and their capabilities. Validation never trusts arbitrary GPIO numbers; it checks pin support against this board profile.

GPIO Validation

GpioManager is responsible for:

  • ensuring module IDs are unique,
  • ensuring pins are not assigned to multiple modules,
  • ensuring each binding role is allowed by the module schema,
  • ensuring the board supports the requested usage for that pin.

Module Instances

A module instance is a runtime driver configuration record. It is not the user-facing device. It describes:

  • which module schema to use (module_type_id),
  • which bindings connect schema roles to real GPIO pins,
  • optional display name,
  • driver settings such as auto_off_ms and external_on_triggers.

Logical Devices

A logical device is the public abstraction that the frontend and automation use. It is backed by a runtime module via driver_module_id.

  • Current built-in modules are all single-binding modules.
  • That means each module instance must resolve to exactly one logical device.
  • If a single-device module has no configured logical device, DeviceRegistry::normalize_config() auto-creates one deterministically.

Auto-Provisioned Logical Devices

For single-device modules, the registry can synthesize:

  • id = <module_id>__device
  • device_type_id = default_device_type_id from the module schema
  • display_name = module display_name or the module ID
  • driver_module_id = module.id

Runtime Module Loading

ModuleManager::load() resolves each module instance through its schema, claims GPIO pins, extracts the schema-defined output and trigger bindings, and constructs the runtime driver.

Right now both core:gpio_switch and core:gpio_output resolve to RelayGpioModule. The difference comes from schema metadata and allowed bindings, not from separate runtime driver implementations.

RelayGpioModule Behavior

The only built-in runtime driver today is RelayGpioModule.

  • It owns one output pin.
  • It may own one optional GPIO trigger input.
  • It tracks binary state On or Off.
  • It supports ON, OFF, and TOGGLE.
  • If auto_off_ms is set, it starts a timer when turned on.
  • If a trigger input is configured, it toggles after debounce.
  • If an external module listed in external_on_triggers turns on, this module toggles.

Module Event Propagation

ModuleManager propagates BinaryStateChanged events across runtime modules. It uses a queue so a change in one module can trigger another module, which can in turn emit more events.

Logical Switch Device Behavior

The current logical device controller is SwitchController.

  • State publication reads the backing module binary state and exposes it as public JSON { "state": "ON" | "OFF" }.
  • Commands are accepted as either plain text or JSON payloads containing a state field.
  • The controller delegates command execution to the driver module.

7. MQTT Contract

MQTT contract code lives in src/mqtt/contract.rs. The topic root is MqttConfig.base_topic, usually nks/home/<device_id>.

Topic Philosophy

  • dev/... topics are public and frontend-facing.
  • mod/... topics are internal/debug/runtime-facing.
  • config/... topics are retained snapshots and schema metadata.
  • status/... topics describe backend availability and identity.

Retained Status And Config Topics

Topic Retained Purpose
status/availability Yes { "state": "online" | "offline" }
status/info Yes Device identity, board identity, firmware version, protocol version, schema version, supported module types, loaded schema packages.
config/board Yes Board info snapshot.
config/resources Yes Effective resource/module snapshot including board pin list and binding usage.
config/devices Yes Effective logical device list.
config/module_types Yes Frontend-facing module type registry snapshot.
config/device_types Yes Frontend-facing device type registry snapshot.

Public Device Topics

Topic Direction Notes
dev/<logical_device_id>/state Firmware publishes Canonical retained public device state topic.
dev/<logical_device_id>/cmd Frontend publishes Canonical public command topic.

Internal Module Topics

Topic Direction Meaning
mod/<module_id>/state Firmware publishes Internal runtime module state.
mod/<module_id>/cmd Internal/manual publisher sends Internal runtime module command path.

Config Command Topics

Topic Request Effect
cmd/get_config { "request_id": "..." } or empty object Re-publishes the retained config/state snapshots.
cmd/validate_resources { "request_id": "...", "resources": ... } Validates the proposed resource config and returns evt/config_result.
cmd/set_resources { "request_id": "...", "resources": ... } Normalizes, validates, saves, reloads runtime, and republishes retained snapshots.

Config Result Topic

evt/config_result is non-retained and carries: 

{
  "request_id": "optional-correlation-id",
  "command": "validate_resources | set_resources | get_config | unknown",
  "ok": true,
  "message": "Human-readable message"
}

Public Device State Payload

{
  "state": "ON"
}

Public Switch Command Payloads

Accepted forms:

ON
OFF
TOGGLE
{
  "state": "ON"
}

8. Provisioning HTTP API

Provisioning is implemented in src/http/mod.rs. It is a staged flow:

  1. Serve the provisioning UI and a cached Wi-Fi scan list.
  2. Test Wi-Fi credentials before saving them.
  3. Test MQTT broker settings before saving them.
  4. Save confirmed settings and reboot.

Main Endpoints

Path Method Purpose
/api/provisioning/status GET Returns AP SSID.
/api/networks GET Returns cached Wi-Fi scan results.
/api/test-wifi POST Schedules Wi-Fi connectivity test.
/api/wifi-status GET Returns Wi-Fi test status.
/api/save-wifi POST Saves tested Wi-Fi credentials.
/api/test-mqtt POST Schedules MQTT broker test.
/api/mqtt-status GET Returns MQTT test status.
/api/save-mqtt POST Saves tested MQTT config and schedules restart.
/api/reset-configuration POST Clears NVS config and reboots.

Provisioning State Tracking

The provisioning controller keeps staged data in memory:

  • pending Wi-Fi test, last successful Wi-Fi test, Wi-Fi test status
  • pending MQTT test, last successful MQTT test, MQTT test status
  • AP SSID and cached scanned networks

Save operations only succeed if the payload exactly matches the last successful test result.

9. Frontend Integration Guide

What The Frontend Should Treat As Stable

  • status/info
  • config/board
  • config/resources
  • config/devices
  • config/module_types
  • config/device_types
  • dev/<id>/state
  • dev/<id>/cmd
  • cmd/get_config, cmd/validate_resources, cmd/set_resources
  • evt/config_result

What The Frontend Should Not Depend On As Public API

  • mod/<module_id>/state
  • mod/<module_id>/cmd
  • driver names such as relay_gpio
  • ESP-specific Wi-Fi or provisioning internals

Suggested Frontend Data Model

  1. Load retained snapshots on connect by subscribing to the full base topic subtree or by calling cmd/get_config.
  2. Build module-type forms from config/module_types.
  3. Build device-type UI and rule/automation menus from config/device_types.
  4. Render configured modules from config/resources.module_instances.
  5. Render public user-facing devices from config/devices.device_instances.
  6. Render live logical device state from dev/<id>/state.

Important Frontend Assumptions That Are Safe Today

  • Current built-in modules are single-channel and single-device.
  • Logical devices map to modules through driver_module_id.
  • Current switch-style devices expose one state field: state.
  • Switch commands are ON, OFF, and TOGGLE.

How To Create Or Update Resources From A Frontend

  1. Construct a full ResourceConfig payload with version, module_instances, and optional explicit device_instances.
  2. Publish it to cmd/validate_resources.
  3. Wait for evt/config_result with ok = true.
  4. Publish the same payload to cmd/set_resources.
  5. Refresh from retained config topics after success.
You do not have to explicitly create device_instances for built-in single-device modules if you are comfortable with the backend's deterministic auto-provisioning behavior. The backend will synthesize them when missing.
